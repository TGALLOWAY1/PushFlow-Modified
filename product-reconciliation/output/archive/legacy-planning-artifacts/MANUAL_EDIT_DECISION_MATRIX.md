# Manual Edit Decision Matrix

This matrix converts the already-settled V3 policy into explicit default behavior for manual edits, analysis, generation, and candidate acceptance.

| move sound to another pad | Active layout pad assignment | Hard Preserve | Replace the prior preserved pad binding with a new preserved pad binding. | Yes. | No, not by default. | The old pad and new pad must be shown as an explicit placement change; later conflicting candidates must be flagged before acceptance. | Layout-scoped, saved with project. | This clears the prior preserved placement and creates a new one. |

## Short Summary of the Preservation Contract

The optimizer must always preserve explicit pad placements and pad finger locks in the active layout unless the user explicitly clears or replaces them. Those are the binding manual edits.

The optimizer may reinterpret `Natural Hand Pose` and sound-level hand/finger preferences because those are guidance, not rules. It should score against them, try to honor them, and explain when it does not.

Per-event manual overrides and analysis-only hide/mute actions are local testing tools. They are valid for verification and debugging, but they are not part of the canonical preserved layout unless the user performs a separate explicit promotion or replacement action.

Metadata and organization edits such as renaming, recoloring, and grouping are not solver constraints, but they must still propagate consistently everywhere and must never be silently changed by optimization.

None at the reconciliation-policy level. This matrix closes the remaining open policy gap; the remaining work is implementation and domain-model drafting, not additional product reconciliation.
